Architecture

How the backend flow works

The UI only triggers the flow. The important work happens asynchronously in the backend.

01 Order API POST /orders receives the request.
→
02 DB + Outbox Order and event are stored together.
→
03 RabbitMQ Relay publishes order.created.
→
04 Inventory Stock is reserved or rejected.
↔

Order creation

The API creates an order and stores the event that must be published.

⇢

Async processing

RabbitMQ decouples order creation from inventory reservation.

✓

Stock decision

Inventory returns either reserved or failed depending on available stock.

●

Final status

Order is updated to CREATED or FAILED after the SAGA response arrives.

Backend concepts

              
What this demo is proving

              
Each card maps to a real backend problem solved by the project.

            


            

              

                S
                
SAGA

                
Order coordinates a multi-step flow and resolves the final state after Inventory responds.

              

              

                O
                
Outbox

                
The order and the event are persisted in the same transaction before publishing.

              

              

                R
                
RabbitMQ

                
Messages are routed through explicit keys with manual Ack/Nack in consumers.

              

              

                I
                
Idempotency

                
Consumers tolerate duplicate deliveries without applying the same business effect twice.

              

              

                N
                
No overselling

                
Stock reservation is concurrency-safe using an atomic PostgreSQL update.
